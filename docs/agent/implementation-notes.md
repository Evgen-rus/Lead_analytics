# Implementation notes: очередь длительных задач и реальный прогресс

## Утверждённые решения

- UI показывает фактический прогресс «N из M» без ETA.
- Задача продолжает выполняться на сервере после обновления или закрытия страницы.
- Первый VPS — общее пространство до 10 пользователей, без учётных записей.
- Одновременно выполняется одна задача; остальные ожидают в очереди.
- Обычные файлы — около 5 000 строк, редкие — до 20 000.
- VPS — утверждённая среда; после успеха удаляются исходные и промежуточные файлы, готовые отчёты сохраняются.

## Отклонения от плана

### D1. Локальный worker вместо отдельного брокера очереди

- Причина: на первом VPS требуется одна задача одновременно, а текущие зависимости не содержат брокера или отдельного worker-сервиса.
- Влияние: задача переживает обновление страницы, но автоматическое восстановление после перезапуска самого backend остаётся отдельной задачей.
- Обратимость: высокая; SQLite-таблица заданий и API статуса пригодны для последующей замены runner на отдельный worker.
- Что сделано: добавлены таблица `processing_jobs`, одиночный daemon-worker и API опроса статуса.

### D2. Оптимизация пакетного сопоставления и записи Excel

- Причина: профилирование реальных файлов на 12 119 и 12 503 строках показало
  161 секунду работы: 78 секунд занимало повторное построение индексов дублей, ещё
  41 секунду — двойная запись и оформление Excel.
- Что сделано: индекс по каждому ключу теперь строится одной сортировкой с сохранением
  правила выбора (последняя дата, иначе последняя строка); наборы ключей и дублей
  рассчитываются заранее. Excel оформляется в памяти до единственного сохранения,
  без повторного открытия книги.
- Влияние: сопоставление тех же файлов занимает 12,48 секунды. Содержимое листов
  результата совпадает с прежним; в листе «Проверка» отличаются только пути
  входных файлов разных запусков.
- Обратимость: высокая; формат и контракт отчёта не менялись.

## Найденные edge cases

- Очередь не должна потерять задание, добавленное на границе остановки worker’а; после освобождения lock повторно проверяются ожидающие задания.
- После обновления страницы UI не может автоматически восстановить введённые поля следующего этапа, но показывает прогресс активного задания и ссылку на готовый отчёт.

## Консервативные допущения

- Общая очередь достаточно прозрачна для общей рабочей области без учётных записей.
- Готовый отчёт сопоставления считается отчётом, а не промежуточным файлом, поэтому сохраняется.

## Закрытые неизвестные

- Контракт прогресса, очередь, параллелизм, профиль файлов, VPS и хранение данных утверждены пользователем.
- Главные узкие места сопоставления на репрезентативной выгрузке и безопасный путь
  оптимизации без изменения правил выбора дублей.

## Оставшиеся неизвестные

- Результаты нагрузочного замера, по которым выбирается точная конфигурация VPS.
- Политика хранения готовых отчётов после периода «пока не удалять».
